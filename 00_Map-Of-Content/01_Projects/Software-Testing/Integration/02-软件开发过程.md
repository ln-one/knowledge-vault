# 02-软件开发过程

## 1. 软件产品的组成部分

### PDF重点：软件的生存期（考试必考）

**软件生存期的7个阶段**：
1. **计划** - 确定项目目标和可行性
2. **需求分析** - 明确用户需要什么功能
3. **设计** - 设计软件的架构和模块
4. **程序编写** - 实际编码实现功能
5. **测试** - 验证软件是否符合要求
6. **运行** - 软件正式投入使用
7. **维护** - 修复问题和功能升级

### PDF重点：软件测试的生命周期（考试重点）

**测试生命周期的5个阶段**：

1. **计划阶段**
   - 任务：安排进度、分配资源、人员及确认测试的起始点和结束点

2. **设计阶段**
   - 任务：尽可能发现系统的Bug,评估系统的性能

3. **开发阶段**
   - 任务：准备测试脚本、测试数据自动生成、测试流程自动化

4. **执行阶段**
   - 任务：执行测试用例

5. **评估阶段**
   - 任务：对测试结果和预期输出进行比较，通过各种量化方法，对测试的有效性及结果的可信度提供量化的依据

### PDF重点：测试用例的分类

**测试用例的4种类型**：
1. **纯数据型测试用例** - 简单的输入输出数据
2. **文件型测试用例** - 涉及文件操作的测试
3. **操作序列型测试用例** - 一系列操作步骤
4. **程序型测试用例** - 复杂的程序逻辑测试

### PDF重点：测试用例设计标准

**ANSI/IEEE829-1983测试用例设计标准**：
- **目的** - 测试用例的目标
- **输入数据** - 测试所需的输入
- **操作步骤** - 具体的执行步骤
- **期望结果** - 预期的输出结果
- **前置条件** - 测试执行前的条件
- **测试数据** - 测试所需的数据
- **测试环境** - 测试执行环境

**测试用例 = 输入数据 + 操作步骤 + 期望结果**

### PDF重点：良好测试用例的特征

1. **可以最大程度地找出软件隐藏的缺陷**
2. **可以最高效率的找出软件缺陷**
3. **可以最大程度地满足测试覆盖要求**
4. **既不过分复杂、也不能过分简单**
5. **使软件缺陷的表现可以清楚的判定**
6. **待查的输出结果或文件必须尽量简单明了**
7. **不包含重复的测试用例**
8. **测试用例内容清晰、格式一致、分类组织**

### 1.1 *软件产品需要多少投入*

> *可交付的部分（deliverable）*：在软件行业中，用于描述制造出来并交付他人的软件产品组件的术语。

*软件产品的主要组成部分：*

1. *客户需求* - 明确用户期望和业务目标
2. *产品说明书* - 开发团队的协定和规范
3. *进度表* - 项目时间管理和里程碑
4. *软件设计文档* - 技术架构和实现方案
5. *测试文档* - 质量保证和验证材料

### 1.2 *常用软件设计文档清单*

- *结构文档* - 系统架构说明
- *数据流图* - 又称泡泡图（bubble chart）
- *状态转换图* - 系统状态变化描述
- *流程图* - 业务流程和逻辑流程
- *代码注释* - 代码级别的说明文档

### 1.3 *重要的测试提交清单*

*测试员也要类似于程序员写文档--测试文档：*

- *测试计划（test plan）* - 测试策略和范围规划
- *测试用例（test case）* - 具体的测试执行步骤
- *缺陷报告（bug report）* - 问题记录和跟踪
- *测试工具和自动测试（test tool and automation）* - 测试效率提升
- *度量、统计和总结（metric, statistic, summary）* - 测试结果分析

### 1.4 *软件产品的完整性*

*重要认识*：当产品打包分发时，分发的不仅仅是代码，许多*支持*也包含在内，*文档*等支持也要测试。

## 2. 软件项目团队成员

### 2.1 *核心团队角色*

- *项目经理、程序经理或者监制人员* - 驱动整个项目
- *架构师或者系统工程师* - 技术架构设计
- *程序员、开发人员或者代码制作者* - 代码实现
- *测试员或质量保证（Quality Assurance，QA）* - 质量控制
- *技术作者、用户协助专员、用户培训专员、手册编写员或者文案专员* - 文档和用户支持
- *配置管理员或构建员* - 版本和构建管理

### 2.2 **团队协作的重要性**

**在软件开发过程中，不同角色需要密切协作：**
- **主持人、作者、记录员** - 评审过程中的关键角色
- **内审员、技术专业人员** - 质量保证专家
- **用户代表** - 需求验证和用户体验

## 3. 软件开发生命周期模式

### 3.1 *大爆炸模式*

*特点*：
- 优点是简单
- 计划、进度安排和正规开发过程几乎没有
- 所有精力都花在开发软件和编写代码上

*测试角色*：测试的工作就是报告发现的问题让客户知道

### 3.2 *边写边改模式*

*特点*：
- 非正规说明书
- 编码、修改，直到成功或是放弃

*适用场景*：
- 可以迅速展现成功
- 适合意在快速制作而且用完就扔的小项目
- 例如原型范例和演示程序

### 3.3 *瀑布模式*

**核心思想**：采用瀑布模式的项目从最初的构思到最终产品要经过一系列的步骤。每一个步骤结束时，项目小组组织审查，并决定是否进入下一步。

**三个重要特点**：
1. **瀑布模式非常强调产品的定义** - 开发或者代码只是其中单独的一块
2. **瀑布模式各步骤是分立的，没有分叉**
3. **瀑布模式无法回溯**

**目标**：在编写代码之前解决所有的未知问题并明确所有细节

**测试特点**：此模式的测试往往放在最后，导致成本可能会较高

### 3.4 螺旋模式


**总体思想**：一开始不必详细定义所有细节，从小开始，定义重要功能，努力实现这些功能，接受客户反馈。

**每一次循环包含6个步骤**：
1. 确定目标、可选方案和限制条件
2. 明确并化解风险
3. 评估可选方案
4. 当前阶段开发和测试
5. 计划下一阶段
6. 确定进入下一阶段的方法

### 3.5 敏捷软件开发


**敏捷软件开发（Agile Software Development）**：容易偏移主题而造成混乱，但强调适应性和快速响应变化。

#### 3.5.1 敏捷宣言的价值观


敏捷开发体现了特定的价值观，强调：
- **个体和互动** 胜过流程和工具
- **工作的软件** 胜过详尽的文档
- **客户合作** 胜过合同谈判
- **响应变化** 胜过遵循计划

#### 3.5.2 敏捷原则要点


1. 尽早和持续地交付有价值的软件来满足客户
2. 欢迎需求变更——即使是在项目开发后期
3. 要不断交付可用的软件，周期从几周到几个月不等，且越短越好
4. 项目过程中，业务人员与开发人员必须在一起工作
5. 要善于激励项目人员，给他们以所需要的环境和支持
6. 面对面的交谈是最有效的沟通方法
7. 可用的软件是衡量进度的主要指标
8. 敏捷过程提倡可持续的开发
9. 对技术的精益求精、对设计的不断完善将提升敏捷性
10. 简单——尽最大可能减少不必要的工作
11. 最佳的架构、需求和设计出自于自组织的团队
12. 团队要定期反省如何能够做到更有效

## 4. 软件测试在开发过程中的作用

### 🔥 PDF重点：软件测试的生命周期（考试重点）

**测试生命周期的5个阶段**：

1. **计划阶段**
   - 任务：安排进度、分配资源、人员及确认测试的起始点和结束点
   > 📅 *项目管理*：就像制定作战计划，明确时间、人员、资源

2. **设计阶段**
   - 任务：尽可能发现系统的Bug，评估系统的性能
   > 🎨 *测试设计*：设计各种测试场景，就像导演设计剧本

3. **开发阶段**
   - 任务：准备测试脚本、测试数据自动生成、测试流程自动化
   > ⚙️ *工具准备*：准备测试工具和环境，就像厨师准备食材

4. **执行阶段**
   - 任务：执行测试用例
   > 🏃 *实际测试*：按计划执行测试，发现和记录问题

5. **评估阶段**
   - 任务：对测试结果和预期输出进行比较，通过各种量化方法，对测试的有效性及结果的可信度提供量化的依据
   > 📊 *结果分析*：分析测试效果，就像医生分析检查报告

### 📝 测试用例的分类（PDF补充）

**测试用例的4种类型**：
1. **纯数据型测试用例** - 简单的输入输出数据
   > 💾 *数据测试*：如测试计算器的加减乘除
2. **文件型测试用例** - 涉及文件操作的测试
   > 📁 *文件操作*：如测试文档的保存和读取
3. **操作序列型测试用例** - 一系列操作步骤
   > 🔄 *流程测试*：如测试用户注册登录流程
4. **程序型测试用例** - 复杂的程序逻辑测试
   > 💻 *程序测试*：如测试复杂的业务逻辑

### 4.1 不同开发模式下的测试策略


#### 4.1.1 传统开发模式
- **重流程** - 严格按照既定流程执行
- **阶段性明确** - 各阶段界限清晰
- **测试团队独立性** - 独立的测试组织
- **测试人员职责细化** - 专业分工明确
- **关注需求/测试文档** - 文档驱动
- **严谨、规范** - 标准化程度高
- **缺陷预防** - 预防性质量管理

#### 4.1.2 敏捷开发模式
- **以人为本，强调个人技能**
- **TDD、ATDD** - 测试驱动开发
- **持续测试** - 持续集成和持续测试
- **侧重产品的测试** - 产品质量导向
- **ET、TA** - 探索式测试和测试自动化
- **整个团队对测试负责** - 全员质量意识
- **拥抱变化** - 适应性强
- **批判性思维能力** - 独立思考
- **自我学习能力** - 持续改进

### 4.2 测试左移和测试右移


#### 4.2.1 测试左移
**含义**：不仅让开发人员做更多的测试，而且需要做需求评审、设计评审，验收测试驱动开发（ATDD）

**具体实践**：
- 需求评审、设计评审等
- 测试计划、测试设计可以在更早时候开始
- TDD、ATDD

#### 4.2.2 测试右移
**含义**：在线测试（Test in Production，TiP），包括在线性能监控与分析、A/B测试和日志分析等

**特点**：
- 高度的可视化、集成化的全过程监控
- 数据驱动质量：客户质量实时反馈
- 可以和现在流行的DevOps联系起来

## 5. 软件开发过程中的质量保证

### 5.1 质量保证的重要性


软件开发不仅仅是编写代码，质量保证贯穿整个开发生命周期：
- 需求阶段的质量保证
- 设计阶段的质量保证  
- 编码阶段的质量保证
- 测试阶段的质量保证
- 维护阶段的质量保证

### 5.2 文档质量的重要性


**认识要点**：
- 文档是软件产品不可分割的一部分
- 文档质量直接影响软件质量
- 文档需要和代码一样进行测试和维护
