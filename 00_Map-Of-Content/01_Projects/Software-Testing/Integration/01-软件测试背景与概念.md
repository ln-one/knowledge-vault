# 01-软件测试背景与概念

## 1. 软件缺陷的真实案例

### 1.1 **现代软件缺陷实例**

- **奥运票务中心、12306的道歉** - 系统负载能力不足
- **动车事故** - 安全系统失效
- **各类产品缺陷**：
  - 输入格式无法正常显示
  - Markdown语法支持不完整
  - 缺陷修改功能异常
  - 本地化功能切换问题
  - 安全性验证缺失

> **重要观点**：软件产品形式越来越多，系统越来越复杂，会存在各种各样的问题，而这需要通过软件测试来发现这些问题。

### 1.2 *历史软件错误案例*

*补充了解的经典案例：*
1. *迪士尼的狮子王* - 动画制作软件缺陷
2. *英特尔奔腾浮点除法缺陷* - 硬件计算错误
3. *美国航天局火星极地登陆者号探测器* - 导航系统失败
4. *爱国者导弹防御系统* - 时间计算精度问题
5. *千年虫问题* - 日期格式处理缺陷
6. *危险的预见：病毒嵌入在JPEG格式图片* - 安全漏洞

## 2. 软件缺陷的定义与特征

### 2.1 **软件缺陷的官方定义**

**IEEE国际标准729定义**：软件缺陷就是软件产品中所存在的问题，最终表现为用户所需要的功能没有完全实现，不能满足或不能全部满足用户的需求。

**软件测试的定义与目的**（PDF重点）：使用人工或自动手段来运行或测定某个系统的过程，其目的在于检验它是否满足规定的需求或是弄清预期结果与实际结果之间的差别。

**产品说明书（product specification）**：软件开发小组的一个协定，简称为说明（spec）或产品说明（product spec）。

### PDF重点：软件的生存期
计划、需求分析、设计、程序编写、测试、运行和维护

### PDF重点：软件测试的六条基本原则（考试必考）

1. **穷尽测试是不可能的**
2. **测试工作具有创造性，但很困难**
3. **测试旨在防止错误的发生**
4. **测试是有风险的**
5. **测试需要有计划性**
6. **测试需要有独立性**

### PDF重点：软件测试的局限性

1. **输入/状态空间的无限性**
2. **故障巧合性**
3. **系统缺陷的不确定性**

### PDF重点：软件测试分类

软件测试分为模块测试、集成测试、确认测试和系统测试。
- **模块测试**：检验每个模块是否能单独工作，从而发现模块的编码问题和算法问题
- **集成测试**：将多个模块连接起来以检验概要设计中对模块之间接口设计的问题
- **确认测试**：以需求规格说明书中的规定作为检验尺度，发现需求分析的问题
- **系统测试**：将开发的软硬件和其他因素综合起来进行全面的检验

### PDF重点：软件测试学科发展历程

**从测试思想导向划分为4个阶段**：
1. **1957～1978年**：以功能验证为导向，测试是证明软件是正确的（正向思维）
2. **1978～1983年**：以破坏性为导向，测试是为了找到软件中的错误（逆向思维）
3. **1983～1987年**：以质量评估为导向，测试是提供产品的评估和质量度量
4. **1988年起**：以缺陷预防为导向，测试是为了展示软件符合设计要求，发现缺陷、预防缺陷

**更好的阶段划分（3个阶段）**：
1. **初级阶段（1957～1971）**：测试通常被认为是对产品进行事后检验，缺乏有效的测试方法
2. **发展阶段（1972～1982）**：1972年第一次关于软件测试的正式会议，促进了软件测试的发展
3. **成熟阶段（1983到现在）**：国际标准Std 829-1983，形成一门独立的学科和专业，成为软件工程学科中的一个重要组成部分

### PDF重点：验证和确认（V&V）

**Test = Verification + Validation**

**Verification**：Are we building the product right？
- 是否正确地构造了软件？即是否正确地做事
- 验证开发过程是否遵守已定义好的内容
- 验证产品满足规格设计说明书的一致性

**Validation**：Are we building the right product？
- 是否构造了正是用户所需要的软件？即是否正在做正确的事
- 验证产品所实现的功能是否满足用户的需求

### PDF重点：静态测试和动态测试

**静态测试**：需求评审、设计评审、代码评审
**动态测试**：通过运行程序来发现软件缺陷

**Verification = Review + Analysis + Test**
1. 穷尽测试是不可能的
2. 测试工作具有创造性，但很困难
3. 测试旨在防止错误的发生
4. 测试是有风险的
5. 测试需要有计划性
6. 测试需要有独立性

### PDF重点：软件测试的局限性
1. 输入/状态空间的无限性
2. 故障巧合性
3. 系统缺陷的不确定性

### PDF重点：软件测试分类
软件测试分为模块测试、集成测试、确认测试和系统测试。
- 模块测试检验每个模块是否能单独工作，从而发现模块的编码问题和算法问题
- 集成测试将多个模块连接起来以检验概要设计中对模块之间接口设计的问题
- 确认测试则应以需求规格说明书中的规定作为检验尺度，发现需求分析的问题
- 系统测试将开发的软硬件和其他因素综合起来进行全面的检验

### 2.2 **软件缺陷的判定标准**

至少满足下列**5个规则**之一才称发生了一个软件缺陷（software bug）：

1. **软件未实现产品说明书要求的功能**
2. **软件出现了产品说明书指明不应该出现的错误**
3. **软件实现了产品说明书未提到的功能**
4. **软件未实现产品说明书虽未明确提及但应该实现的目标**
5. **软件难以理解、不易使用、运行缓慢或者--从测试员的角度看--最终用户会认为不好**

> **注意**：测试要客观评价，并非所有测试发现的缺陷都要修改。

### 2.3 **软件缺陷的表现形式**

**从产品内部看**：软件产品开发或维护过程中所存在的错误、毛病等各种问题

**从外部看**：系统所需要实现的某种功能的失效或违背

**主要类型/现象**：
- **功能、特性没有实现或部分实现**
- **设计不合理，存在缺陷**
- **实际结果和预期结果不一致**
- **运行出错，包括运行中断、系统崩溃、界面混乱**
- **数据结果不正确、精度不够**
- **用户不能接受的其他问题，如存取时间过长、界面不美观**

## 3. 软件缺陷产生的原因

### 3.1 **主要原因分析**

令人惊讶的发现：
- **最大原因**：产品说明书（主要是不够全面，经常更改）
- **第二大来源**：设计（随意、易变、沟通不足）
- **第三**：编码缺陷
- **其他**：误解（将本来正确的当成缺陷）

### 3.2 **具体产生因素**

#### 3.2.1 技术问题
- 算法错误
- 语法错误
- 计算和精度问题
- 接口参数传递不匹配

#### 3.2.2 团队工作
- 误解、沟通不充分
- 人与人的交流比写程序困难得多

#### 3.2.3 软件本身
- 文档错误
- 用户使用场合（user scenario）
- 时间上不协调、或不一致性所带来的问题
- 系统的自我恢复或数据的异地备份、灾难性恢复等问题

### 3.3 **项目管理问题**

- **项目没有被很好地理解；计划不周，最终导致进度拖延**
- **没有充分的文档资料**
- **软件可靠性缺少度量的标准，质量无法保证**
- **软件难以维护、不易升级**

## 4. 软件缺陷的修复成本

### 4.1 **成本增长规律**

**修复软件缺陷的费用是随着时间推移而增加的，费用可呈指数级增长。**

> **关键原则**：软件测试员的目标是**尽可能早**地找出软件缺陷，并确保其得以**修复**。

### 4.2 *"修复"的含义*

"修复"缺陷并非指一定要修正软件。可以是指：
- 在用户手册中增加一段注释
- 为用户提供特殊的培训
- 其他补偿措施

## 5. Bug一词的历史由来

### 5.1 **Bug一词的历史由来**

**时间**：1945年9月9日，一个炎热的下午

**地点**：第一次世界大战时建造的老建筑机房，没有空调，所有窗户都敞开着

**人物**：Grace Hopper正领导着一个研究小组

**事件**：研制一台称为"MARK II"的计算机时，机器突然死机。经检查发现是一只飞蛾卡在继电器中导致的故障。

**这就是"Bug"一词在计算机领域的由来，从此软件缺陷被称为"Bug"。**

## 6. 质量与缺陷的关系

### 6.1 **缺陷是质量的对立面**

**要了解什么是缺陷（defect），就必须清楚"质量（Quality）"概念，因为缺陷是相对质量而存在的。**

**违背了质量、违背了客户的意愿，不能满足客户的要求，就会引起缺陷或产生缺陷。**
