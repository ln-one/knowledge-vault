# 03-软件测试实质

## 1. 软件测试的基本原则

### PDF重点：软件测试的六条基本原则（考试必考）

1. **穷尽测试是不可能的**
2. **测试工作具有创造性，但很困难**
3. **测试旨在防止错误的发生**
4. **测试是有风险的**
5. **测试需要有计划性**
6. **测试需要有独立性**

### 软件测试的局限性（PDF补充）

1. **输入/状态空间的无限性**
2. **故障巧合性**
3. **系统缺陷的不确定性**

### 1.1 完全测试程序是不可能的


**四个不可能的原因**：
1. **输入量大** - 输入组合数量庞大
2. **输出结果太多** - 输出状态复杂多样
3. **软件执行路径太多** - 程序逻辑路径数量巨大
4. **软件说明书不符合用户需求** - 需求理解偏差

> **重要结论**：穷举测试在实际中是不可行的，必须采用有效的测试策略。

### 1.2 软件测试有风险


测试过程中存在各种风险，需要进行风险评估和管理。测试不能保证软件100%无缺陷。

### 1.3 测试无法显示潜伏的软件缺陷


测试只能发现已经暴露的缺陷，无法保证发现所有潜在问题。

### 1.4 找到的缺陷越多，软件缺陷越多

### PDF重点：黑盒测试与白盒测试的完整定义

**黑盒测试**：又称功能测试、数据驱动测试或基于规格说明的测试，被测程序被当作打不开的黑盒，无法了解其内部结构和内部特性，测试者只知道该程序输入和输出之间的关系，或是程序的功能，他必须依靠能够反应这一关系和程序功能的需求规格说明书考虑确定测试用例和推断测试结果的正确性。

**白盒测试**：又称结构测试、逻辑驱动测试或基于程序的测试。测试者可以看到被测的源程序，他可利用分析程序的内部构造，并且依据其内部结构设计测试用例，此时测试者完全不顾程序的功能。在进行单元测试时大多采用白盒测试，在确认测试或系统测试时大多采用黑盒测试。

### PDF重点：回归测试

**回归测试**：将已经测试过的测试用例拿来重新进行测试，以检查错误是否真的修改以及是否引入新的错误。

**分类**：
- **改错性回归测试** - 修复缺陷后的验证测试
- **增量性回归测试** - 新功能添加后的验证测试

**重用策略**：
- **Retest-all Strategy** - 重新执行所有测试用例
- **Selective Retest Strategy** - 选择性重新执行相关测试用例

### PDF重点：软件审查的步骤

软件审查包括以下步骤：
1. **制定计划** - 确定审查目标和范围
2. **预审** - 初步检查审查材料
3. **准备** - 准备审查会议材料
4. **审查会** - 正式审查会议
5. **审查返工** - 根据审查结果进行修改
6. **终审** - 最终确认审查结果

### PDF重点：自动测试过程

自动测试过程包括以下步骤：
1. **选择测试用例** - 确定需要自动化的测试用例
2. **准备测试数据+选择测试步骤，设计测试脚本** - 编写自动化脚本
3. **设置测试环境** - 配置测试运行环境
4. **重复执行测试用例** - 自动化执行测试
5. **清除测试环境** - 清理测试环境
6. **总结测试结果** - 分析和报告测试结果


**影响因素**：
- **程序员心情** - 开发状态影响代码质量
- **程序员的习惯** - 编程习惯决定缺陷模式
- **某些缺陷是冰山一角** - 表面缺陷往往暗示更深层问题

### 1.5 杀虫剂怪事


长期使用相同的测试方法和测试用例，会导致测试效果递减，就像杀虫剂使用久了害虫会产生抗药性一样。

**应对策略**：
- 定期更新测试用例
- 采用不同的测试方法
- 引入新的测试技术

### 1.6 并非所有的软件缺陷都要修复


**不修复的原因**：
1. **没有足够的时间** - 项目进度限制
2. **不算真正的软件缺陷** - 误报或需求理解偏差
3. **修复的风险太大** - 可能引入新的问题
4. **不值得修复** - 成本效益分析结果

### 1.7 缺陷难以定义


缺陷的定义往往依赖于上下文和用户期望，同一个现象在不同情况下可能被认为是缺陷或正常行为。

### 1.8 产品说明书没有最终版本


需求和规格说明书在开发过程中会不断变化，测试需要适应这种变化。

### 1.9 软件测试员在产品小组不受欢迎


**原因**：软件测试员的工作是检查和批评同事的工作。

**应对策略**：
- **早点找出缺陷** - 越早发现成本越低
- **控制情绪** - 保持专业和客观
- **不要总是报告坏消息** - 平衡正面和负面反馈

## 2. 软件测试的术语和定义

### 2.1 精确和准确


- **精确（precision）** - 测量结果的一致性和重复性
- **准确（accuracy）** - 测量结果与真实值的接近程度

这两个概念在测试中都很重要，需要同时追求精确性和准确性。

### 2.2 确认和验证（V&V）


#### 2.2.1 基本定义
- **确认（verification）** - 保证软件符合**产品说明书**的过程
  - "Are we building the product right？" - 是否正确地构造了软件？
  - 验证开发过程是否遵守已定义好的内容
  - 验证产品满足规格设计说明书的一致性

- **验证（validation）** - 保证软件满足**用户**要求的过程
  - "Are we building the right product？" - 是否构造了正是用户所需要的软件？
  - 验证产品所实现的功能是否满足用户的需求

#### 2.2.2 V&V关系


**Test = Verification + Validation**

软件测试是由"验证（Verification）"和"有效性确认（Validation）"活动构成的整体。

### 2.3 质量和可靠性


**可靠性仅仅是质量（quality）的一个方面。**

质量是一个更广泛的概念，包括：
- 功能性
- 可靠性
- 易用性
- 效率
- 可维护性
- 可移植性

### 2.4 测试和质量保证


#### 2.4.1 测试（Testing）
**软件测试员的目标**：尽可能地**找出软件缺陷**，并确保缺陷得以修复

#### 2.4.2 质量保证（Quality Assurance, QA）
**软件质量保证人员的主要目标**：创建和执行改进软件开发过程并防止软件缺陷发生的**标准和方法**

## 3. 软件测试的定义

### 3.1 软件测试的正面性定义


#### 3.1.1 Bill Hetzel博士的定义（正向思维）
- 软件测试就是为程序能够按预期设想那样运行而建立足够的信心
- "软件测试是一系列活动以评价一个程序或系统的特性或能力并确定是否达到预期的结果"
- 测试是为了验证软件是否符合用户需求，即验证软件产品是否能正常工作

#### 3.1.2 IEEE的定义
- 在特定的条件下运行系统或构件，观察或记录结果，对系统的某个方面做出评价
- 分析某个软件项以发现现存的和要求的条件之差别（即错误）并评价此软件项的特性

### 3.2 软件测试的反面性定义


#### 3.2.1 Glenford J. Myers的定义（反向思维）
- 测试是为了证明程序有错，而不是证明程序无错误
- 一个好的测试用例是在于它能发现至今未发现的错误
- 一个成功的测试是发现了至今未发现的错误的测试

### 3.3 软件测试定义的两面性


软件测试具有双重特性：

**正向思维**：
- 验证软件正常工作
- 评价一个程序或系统的特性或能力并确定是否达到预期的结果
- 在设计规定的环境下运行软件的所有功能，直至全部通过

**逆向思维**：
- 假定软件有错误
- 测试是为发现错误而针对某个程序或系统的执行过程
- 寻找容易犯错误的地方和系统的薄弱环节，试图破坏系统，直至找不出问题

## 4. 静态测试和动态测试

### 4.1 基本概念


#### PDF重点：黑盒测试与白盒测试（考试必考）

#### 黑盒测试
**定义**：又称功能测试、数据驱动测试或基于规格说明的测试，被测程序被当作打不开的黑盒，无法了解其内部结构和内部特性，测试者只知道该程序输入和输出之间的关系，或是程序的功能，他必须依靠能够反应这一关系和程序功能的需求规格说明书考虑确定测试用例和推断测试结果的正确性。

**适用场景**：在确认测试或系统测试时大多采用黑盒测试

#### 白盒测试
**定义**：又称结构测试、逻辑驱动测试或基于程序的测试。测试者可以看到被测的源程序，他可利用分析程序的内部构造，并且依据其内部结构设计测试用例，此时测试者完全不顾程序的功能。

**适用场景**：在进行单元测试时大多采用白盒测试

### PDF重点：回归测试

**回归测试**：将已经测试过的测试用例拿来重新进行测试，以检查错误是否真的修改以及是否引入新的错误。

**分类**：
- **改错性回归测试** - 修复缺陷后的验证测试
- **增量性回归测试** - 新功能添加后的测试

**回归测试策略**：
- **Retest-all Strategy** - 重新执行所有测试用例
- **Selective Retest Strategy** - 选择性重新测试

### PDF重点：软件审查的步骤

1. **制定计划** - 确定评审的范围、时间、人员
2. **预审** - 初步检查评审材料
3. **准备** - 评审人员熟悉材料
4. **审查会** - 正式的评审会议
5. **审查返工** - 根据评审意见修改
6. **终审** - 确认修改结果

### PDF重点：自动测试过程

1. **选择测试用例** - 确定哪些测试适合自动化
2. **准备测试数据+选择测试步骤，设计测试脚本** - 编写自动化脚本
3. **设置测试环境** - 配置测试环境
4. **重复执行测试用例** - 自动化执行测试
5. **清除测试环境** - 清理测试数据
6. **总结测试结果** - 分析测试报告

### 4.1.1 静态测试
**定义**：不执行程序，而是通过人工评审、代码扫描/静态分析发现错误

**主要形式**：
- 需求评审
- 设计评审  
- 代码评审

#### 4.1.2 动态测试
**定义**：通过运行程序来发现软件缺陷

**验证关系**：
**Verification = Review + Analysis + Test**

### 4.2 静态测试的详细内容


#### 4.2.1 产品评审
**评审对象**：需求文档、设计和代码

**价值**：
- 通过软件评审，可以更早地发现需求工程、软件设计等各个方面的问题
- 大大减少大量的后期返工
- 将质量成本从昂贵的后期返工转化为前期的缺陷发现

#### 4.2.2 需求评审
**解决的问题**：
- 不正确的需求认识
- 丢掉的需求点
- 模糊的描述
- 多余的或没意义的需求
- 不一致的理解

**评审标准**：
- 正确性
- 完备性
- 易理解性
- 一致性
- 易测试性
- 易追溯性

#### 4.2.3 设计评审
**目标**：
- 保证需求能在设计中得到准确和完整的表示
- 保证系统架构设计和产品功能规格说明书的质量

**关注点**：
- 系统架构
- 可测试性
- 系统部署

#### 4.2.4 代码静态分析
**两种情况**：
1. **人工检测** - 偏重于编码风格、质量的检验，对设计、代码进行分析
2. **计算机辅助静态分析** - 利用静态分析工具对被测程序进行特性分析

**技术手段**：
- 词法分析（Lexer）
- 语法分析（parser）
- 控制流分析
- 数据流分析
- 符号执行
- 抽象解释
- 值依赖分析

## 5. 软件测试学科的发展历程

### 5.1 按思想导向划分的4个阶段


1. **1957～1978年** - 以功能验证为导向，测试是证明软件是正确的（正向思维）
2. **1978～1983年** - 以破坏性为导向，测试是为了找到软件中的错误（逆向思维）
3. **1983～1987年** - 以质量评估为导向，测试是提供产品的评估和质量度量
4. **1988年起** - 以缺陷预防为导向，测试是为了展示软件符合设计要求，发现缺陷、预防缺陷

### 5.2 按发展程度划分的3个阶段


1. **初级阶段（1957～1971）** - 测试通常被认为是对产品进行事后检验，缺乏有效的测试方法
2. **发展阶段（1972～1982）** - 1972年第一次关于软件测试的正式会议，促进了软件测试的发展
3. **成熟阶段（1983到现在）** - 国际标准Std 829-1983，形成一门独立的学科和专业，成为软件工程学科中的一个重要组成部分

## 6. 软件测试的现代理念

### 6.1 敏捷测试特征


1. 尽早和持续地开展测试
2. 能及时完成对软件质量全面评估
3. 软件本身是测试研究和分析最主要的对象
4. 在满足所要求的质量，测试进行得越快越好
5. 测试人员必须和项目干系人保持密切协作
6. 对测试人员足够信任和尊重
7. 测试计划、设计和执行力求简单
8. 对测试技术精益求精
9. 不断反思，持续优化测试设计

### 6.2 敏捷测试的本质


**敏捷测试 = 持续的质量反馈**

通过持续的质量反馈机制，确保：
- 需求质量
- 设计质量
- 代码质量
- 功能质量
- 非功能特性质量
