# 交互任务分析

## 背景：
任务分析用文档描述如何完成任务，涉及工作流程各方面，用于探讨系统需求和组织数据收集。主要调查现有情形，通过观察访谈收集5W/2H数据。
> 注：5W/2H = Who(谁)、What(什么)、When(何时)、Where(何地)、Why(为什么)、How(如何)、How much(多少)

## 1. 基本概念：

### 1.1. 核心概念：
- **目标（Goal）**：用户期望达到的最终状态
- **任务（Task）**：完成目标的操作序列集合，可分解为子任务  
- **动作（Action）**：达到目标的具体活动，任务最小单元
> 例：目标"借书" → 任务"检索图书" → 动作"输入书名"

### 1.2. 任务分析定义：
记录人们如何完成任务，通过观察访谈收集数据。
**作用**：了解目标、达成方式，建立需求，为系统设计提供用户行为描述。

## 2. 层次化任务分析（HTA）：

### 2.1. HTA定义：
应用最广的任务分析技术，自顶向下分析方法：
- 任务分解为子任务
- 子任务进一步分解
- 组织成"执行次序"
> 注：HTA = Hierarchical Task Analysis，层次化任务分析，考试重点方法

### 2.2. HTA四要素：
- **目标（Goal）**：完成任务后期望的系统状态
- **任务（Task）**：完成目标的操作序列集合
- **规划（Plan）**：子任务执行条件和顺序
- **操作（Operation）**：达到目标的具体活动
> 注：记忆口诀 - GTPO（Goal Task Plan Operation），HTA的基本框架

### 2.3. HTA方法：
**步骤**：具体目标开始 → 添加任务子目标 → 自顶向下结构化 → 流程图符号+细节 → 检查完整性

### 2.4. HTA图形表示：
- **方框**：上层表示子目标，下层表示具体操作
- **计划**：数字编号表示执行顺序
- **线条**：连接任务和子任务
- **结构**：树状，根节点向下分解，节点编号便于管理

## 3. 任务分解：

### 3.1. 定义：
过程的线性描述，确认记录过程，描述步骤、元素、环境因素。目的是将复杂任务拆解为可管理小单元。

### 3.2. 获取信息：
**5W/2H要素**：
- **Why**：任务目的意义
- **Who**：执行者角色
- **What**：所需对象信息
- **When**：时间要求时序
- **Where**：执行环境

**详细信息**：
- **信息流**：输入输出关系
- **工具**：使用的设备系统
- **序列依赖**：执行顺序相互依赖
- **环境条件**：环境要求约束
- **文化约束**：影响执行的文化因素

### 3.3. 示例：安排会议

#### 目标：安排团队会议
#### 计划：根据成员时间预定会议室及设备
#### 信息需求：成员联系信息、会议室时间表、设备使用手续
#### 物理对象：会议室日程表、通讯录、设备签出表

#### 联系方法：
- 电子邮件（首选）- 异步有记录
- 即时信息 - 实时快速
- 面对面 - 直接高效
- 电话 - 实时语音
- 便条 - 物理留言

#### 子目标：
1. 联系团队成员 - 收集时间
2. 确认回应 - 跟踪反馈
3. 调整时间表 - 协调冲突
4. 安排会议室 - 预订场地
5. 安排设备 - 确保技术支持

#### 异常处理：
联系方法失效时备选：检查邮件状态、监控在线情况、直接面谈、电话联系、留便条

## 4. HTA执行次序：

### 4.1. 图书馆借书示例：
```
0. 借书
├── 1. 前往图书馆
├── 2. 检索图书
│   ├── 2.1 访问目录 → 2.2 使用检索屏 → 2.3 输入准则
│   ├── 2.4 找出图书 → 2.5 记录位置
├── 3. 找书架取书
└── 4. 办理借阅手续
```

**执行次序**：
- Plan 0：1→2→3→4（顺序执行）
- Plan 2：2.1→2.2→2.3→2.4→2.5；未查到重新执行2.5（循环）
- 异常：书不在书架上，重新执行2.4→3→4（异常处理）
> 注：Plan编号对应任务编号，0表示总体计划，数字表示执行条件和分支

### 4.2. 安排会议示例：
```
0. 安排会议
├── 1. 列出与会者名单
├── 2. 列出会议限制表  
├── 3. 确定会议时间
│   ├── 3.1 院系日程 → 3.2 个人日程 → 3.3 比较时间 → 3.4 确定时间
├── 4. 输入日程安排
└── 5. 通知与会者
```

**执行次序**：
- Plan 0：1→2→3→4→5
- Plan 3：3.1→3.2→3.3→3.4；找到时间执行3.4，否则重复子序列（迭代特性）

## 5. 终止规则：

任务分析迭代过程，需明确终止条件避免过度分解：

### 5.1. 终止条件：
- **物理操作**：鼠标移动、键盘输入等基本操作（UI设计基本单元）
- **认知决策**：
  - 与外部动作相关→继续分解
  - 纯认知性决策→终止分解
- **实用性**：进一步分解无价值时终止
> 注：终止规则避免过度分解，平衡分析完整性和实用性

### 5.2. 判断标准：
设计相关性、观察可行性、实现复杂度

## 6. 任务金字塔：

### 6.1. 层次结构：
```
总体目标（Why - 战略层）
    ↓
子目标（What - 战术层）  
    ↓
具体操作（How - 操作层）
```

### 6.2. 分解原则：
- 复杂任务逐层分解为简单子任务
- 每层有相应抽象程度
- 子任务完整覆盖上层任务

## 7. HTA应用领域：

### 7.1. 手册和教学：
- 操作手册编写（拆卸清洁武器、软件安装）
- 培训材料开发
- 优势：层次清晰、步骤明确、适合不同技能用户

### 7.2. 需求获取和系统设计：
- 任务分析是需求分析工具（非需求获取本身）
- 完整表达验证需求
- 提供用户行为依据
- 识别真实需求、发现问题点、提供设计参考

### 7.3. 接口设计：
- 菜单设计：按任务层次组织
- 工作流设计：符合用户认知
- 功能布局：按任务逻辑分组
- 界面结构反映任务结构

## 8. 故事板（Storyboard）：

### 8.1. 定义：
用系列图片描述过程或工作流程的可视化方法，类似电影分镜头脚本。

### 8.2. 特点：
- **直观性**：图像展示任务过程
- **时序性**：清晰展示动作顺序
- **情境性**：包含环境工具人物上下文
- **沟通价值**：便于理解交流、发现问题、支持迭代

### 8.3. 应用：
用户体验设计、系统演示、培训材料（设计早期成本低效果好）

## 9. 用例分析（Use Case）：

### 9.1. 用例定义：
描述工作流程的形式化结构化方法，探讨用户与系统交互。由Jacobson(1992)提出，纳入UML标准。

**核心思想**：从用户角度描述系统功能，关注目标非技术实现，提供系统边界定义。
> 注：Use Case是软件工程标准方法，UML统一建模语言的重要组成部分

### 9.2. 参与者（Actor）：
与系统交互的外部实体，包括人、系统、网络、软件。

**会议系统参与者**：
- 协调人（主要用户）
- 会议室（资源）  
- 视听设备（资源）
- 小组成员（次要用户）

**特征**：系统边界外、直接交互、明确角色职责

### 9.3. 用例特点：
- 每个参与者有特定用例（任务或目标）
- 描述短期离散目标，非长期目标
- 涵盖系统使用的各种方法和功能
- 关注"做什么"非"怎么做"
- 参与者定义系统外，用例定义系统内
- 包含循环分支条件结构，覆盖所有交互场景

### 9.4. 用例路径：

#### 基本路径（Basic Flow）：
无错误条件转移路径，完成预定目标，无条件语句。代表理想执行情况，"一切顺利"的标准流程。

#### 可选路径（Alternative Flow）：
基本路径外的其他情况：
- 提前终止
- 方法选择  
- 错误处理
体现系统鲁棒性和灵活性，测试异常处理能力。
> 注：基本路径=主成功场景，可选路径=异常场景，两者结合覆盖所有可能情况

### 9.5. 场景（Scenario）：
用例任一路径的具体实例化，有明确开始结束，联合创建完整用例，是最低层。

#### "安排会议"场景示例：
- **场景1（基本）**：协调者预定房间设备，通知成员，邮件确认参加
- **场景2（可选1）**：房间预定但设备不可用，寻找替代或调整安排
- **场景3（可选2）**：房间设备预定，但成员未回复邮件，其他方式确认

### 9.6. 事件流程示例：
"安排会议"事件流程：
1. 初始联系：查看通信录，群发邮件征询时间
2. 跟踪反馈：检查邮件回复
3. 备选沟通：无回应时使用即时信息、面谈、电话、便条
4. 时间协调：创建可用时间汇总
5. 场地预订：检查会议室安排表并预订
6. 设备预订：检查设备签出表并预订
7. 冲突处理：设备不可用时重新调整需求

**流程特点**：顺序性、条件性、迭代性、适应性

### 9.7. 用例分析图：
**UML符号**：
- 参与者：人形图标
- 用例：椭圆形
- 关系线：直线连接

**关系方向**：
- 参与者→用例：主动使用功能
- 用例→参与者：执行影响参与者

**价值**：系统边界可视化、功能需求覆盖检查、参与者职责划分

## 10. 其他分析方法：

### 10.1. 顺序分析：
关注任务执行时间顺序和步骤流程。
**优点**：时序清晰、步骤明确、逻辑性强（适用明确先后关系任务）
**缺点**：关系模糊、静态视角、复杂度限制（难表现并行交互）
**应用**：操作手册、培训流程、简单任务优化

### 10.2. 协作关系分析：
关注系统元素相互作用和依赖关系。
**特点**：关系导向、并行视角、交互性强（适用多方协作场景）
**分析维度**：角色关系、信息流、依赖关系、职责分工
**价值**：识别瓶颈冲突、优化协作流程、支持团队工具设计

### 10.3. 用户任务一览表：
表格形式描述所有用户及任务。

| 用户角色 | 任务1 | 任务2 | 任务3 |
|---------|-------|-------|-------|
| 角色A   |   ✓   |   ✗   |   ✓   |
| 角色B   |   ✗   |   ✓   |   ✓   |

**价值**：全面性、权限管理、功能覆盖、设计指导（适用多用户多角色系统）
**场景**：需求分析、权限设计、功能划分、测试用例

## 11. 原型设计：

### 11.1. 重要性：
**用户认知局限**：用户难以准确描述需要，看到后才知道不想要什么
**沟通桥梁**：弥合用户需求与技术实现鸿沟，提供具体交互体验

### 11.2. 原型定义：
接近实际产品的模型，用于评估改进技术方案。
**作用**：体验模拟、需求发现、风险降低

### 11.3. 原型分类：

#### 低保真原型：
- 与最终产品差异大，使用不同材料（纸张、纸板）
- 关注功能流程非视觉细节
- **优点**：成本低、快速迭代、专注核心、早期验证
- **适用**：概念验证、需求探索、方案比较
> 注：如PalmPilot木雕原型，测试基本交互概念

#### 高保真原型：
- 接近最终产品，相似材料技术，真实外观交互
- **优点**：真实体验、详细测试、技术验证、演示效果
- **风险**：用户误解、开发陷阱、成本高、过早细化
> 注：如Visual Basic开发的软件原型，多数项目首选

### 11.4. 选择策略：
早期→低保真探索需求；中期→结合使用验证设计；后期→高保真最终验证

## 12. 需求验证：

### 12.1. 必要性：
用户难以准确描述真实需要，需求文档抽象导致理解偏差，设计师与用户认知差异。
原型提供共同参考点，减少偏差。

### 12.2. 验证方法：
用户测试、专家评审、A/B测试、迭代改进

## 小结：

交互任务分析是理解用户工作流程和系统需求的重要方法，为HCI设计提供科学分析框架。

### 核心方法：

1. **层次化任务分析（HTA）**：结构化分解任务，总体目标→具体操作，提供执行次序和条件分支
2. **任务分解技术**：详细描述执行过程，获取5W/2H信息，识别信息流工具环境
3. **用例分析方法**：形式化描述用户系统交互，明确边界角色，覆盖基本路径和异常处理
4. **原型验证技术**：低/高保真原型验证需求，弥合需求实现鸿沟，支持迭代改进

### 应用价值：

**设计指导**：提供用户行为依据，确保功能任务匹配，支持以用户为中心设计
**需求管理**：发现澄清真实需求，提供验证确认手段，降低变更返工风险  
**团队协作**：提供共同理解基础，支持决策科学化可视化，促进有效沟通

### 方法选择：

- 简单任务：任务分解+顺序分析
- 复杂系统：HTA+用例分析
- 多用户系统：协作关系分析+用户任务矩阵
- 创新产品：原型验证+迭代改进

通过系统应用这些方法，设计师能深入理解用户需求，创造满足工作目标的交互系统。
